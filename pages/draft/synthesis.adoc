[NOTE.speaker]
====
On a vu 3 contextes très différents.
Une forge logicielle construite from scratch, un audit ELK, une infra de 350 VMs.
Pourtant, des patterns reviennent. C'est ce qu'on va résumer ici.
====

// ─── DO ──────────────────────────────────────────────────────────────────────

[%notitle]
== Do ✓

[%step]
* Définir un schéma de log *dès le départ*
* ILM et rétention *dès J1*
* Une alerte = un runbook (même minimal)
* Mesurer le MTTR, pas le nombre de métriques
* Alerter sur l'écart à la baseline, pas sur des seuils statiques

[NOTE.speaker]
====
* Ces 5 points reviennent dans les 3 REXs sous des formes différentes
* Le schéma de log : fondation de tout. Sans ça, impossible de corréler.
* ILM dès J1 : le remédier plus tard coûte 10x plus cher.
* Runbook minimal : même "regarder le dashboard X" c'est mieux que rien.
====

// ─── DON'T ───────────────────────────────────────────────────────────────────

[%notitle]
== Don't ✗

[%step]
* Logger en DEBUG en production
* Alerter sur tout — c'est alerter sur rien
* Faire confiance aux estimations de volumétrie
* Déployer un pipeline "générique" qui fait tout
* Monitorer sans inventaire

[NOTE.speaker]
====
* DEBUG en prod : le piège le plus courant, et le plus coûteux en volumétrie
* Alerter sur tout : on l'a vu dans les 3 REXs — le bruit tue l'utilité du monitoring
* Estimations de volumétrie : elles sont toujours fausses. Mesurez.
* Pipeline générique : appelez-le "legacy" dans 6 mois
* Sans inventaire : vous ne savez pas ce que vous ne monitorez pas
====

// ─── CHECKLIST ───────────────────────────────────────────────────────────────

[%notitle]
== Checklist de départ

[%step]
* [ ] Inventaire des sources de logs et métriques
* [ ] Schéma de log défini (champs obligatoires, typage)
* [ ] Politique de rétention par type de donnée
* [ ] Profils de VM / service définis
* [ ] Seuils d'alerte basés sur une baseline
* [ ] Runbook pour chaque alerte P1/P2

[NOTE.speaker]
====
* Cette checklist ne garantit pas un monitoring parfait
* Elle garantit qu'on ne part pas de zéro dans 6 mois
* Chaque item est issu d'une douleur concrète dans l'un des 3 REXs
