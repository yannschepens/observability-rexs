[%step]

* Disponibilité

== Collecteur : fichiers

image::rex1-filebeat.png[1, 60%]

[NOTE.speaker]
====
Problématique : lire le contenu des fichiers pour les stocker.

Solution : Filebeat

* Service externe à Docker
* Beaucoup d'extensions
* Parse le JSON nativement
* Possibilité d'ajouter / modifier les logs
* Renvoie les logs vers beaucoup d'outils différents

Transition : Et pour les logs Docker — la sortie standard.
====

[%notitle]
== Collecteur : Docker

image::rex1-docker-driver.png[1, 60%]

[NOTE.speaker]
====
Problématique : récupérer les logs stdout/stderr Docker.

Docker propose plusieurs drivers : none, json-file, syslog, fluentd, awslogs, splunk...

On a choisi : Docker GELF log driver

* Natif Docker
* Configuration dans les docker-compose
* Renvoie les logs vers un autre outil au format JSON
* Possibilité d'ajouter du contenu

== Agrégateur : Logstash

image::rex1-logstash.png[1, 50%]

[NOTE.speaker]
====
Problématique : pour normaliser les logs, il faut d'abord les centraliser.

Solution : Logstash

* Plusieurs connecteurs d'input

== Stockage : organisation des index

image::rex1-indexes.png[1, 75%]

[NOTE.speaker]
====
Problématique : organiser le stockage pour simplifier l'accès et la maintenance.

Solution : des index par type et origine des logs

* Découpage par mois et années
* Format : `<service>_<env>-<log_file>-YYYY-MM`
* Une centaine d'index par mois
* Nettoyage via un script selon le type :
  ** 1 an pour les logs d'accès
  ** 1 mois pour les logs d'erreurs
  ** 1 mois pour les logs applicatifs

== Exploitation avec Grafana

[NOTE.speaker]
====
On a la stack. Voyons ce qu'on fait concrètement avec Grafana comme point central.
Grafana agrège tout : logs via Elasticsearch, métriques via Zabbix, et permet l'alerting avec authentification.
Kibana reste l'outil de navigation fine dans les logs — mais Grafana est le tableau de bord opérationnel.
====

[%notitle]
== Navigation dans les logs : Kibana

Recherches sauvegardées

[%step]
* Access logs par outil, par adresse IP, codes 500
* Logs d'erreur par outil et par environnement
* Lien direct depuis les graphiques Grafana → Kibana
* Regroupement par point d'accès : admin vs utilisateurs

[NOTE.speaker]
====
* Kibana : l'outil de navigation fine, pour explorer et filtrer les logs
* Les recherches sauvegardées sont sa vraie valeur — on ne retape pas les mêmes filtres
* Lien Grafana → Kibana : une alerte Grafana pointe vers la recherche Kibana correspondante
* Kibana pas exposé sur internet (pas d'auth en version gratuite) → usage interne uniquement

Transition : Grafana, lui, est le point central du monitoring.
====

[%notitle]
== Dashboards et workflows Grafana

[%step]
* Erreurs 500, nombre de requêtes (avec/sans bots), volume de logs d'erreur
* Alertes sur les 500 et sur l'absence de logs (via pings Zabbix)
* Cartographie des services — plugins Grafana
* Carte des accès étrangers — GeoIP Logstash + worldMap Grafana
* Couplage avec Zabbix : Grafana surveille Zabbix qui surveille Grafana

[NOTE.speaker]
====
* Grafana devient le point central du monitoring — tout y converge
* L'alerte sur l'absence de logs est une idée simple mais puissante : si un service arrête d'écrire des logs, c'est peut-être qu'il ne fonctionne plus
* Pour ça on s'appuie sur les pings Zabbix : si Zabbix confirme que la VM est up mais qu'on ne voit plus de logs → le service est planté
* Grafana surveille Zabbix qui surveille Grafana — surveillance mutuelle pour éviter l'angle mort

[%notitle]
== Tracking des utilisateurs

Qui va où et quand ?

[%step]
* Les utilisateurs passent tous par GitLab pour s'authentifier
* Les logs GitLab contiennent : nom d'utilisateur + adresse IP
* Les access logs contiennent les adresses IP
* Croisement → on sait qui accède à quoi et quand

[NOTE.speaker]
====
* Sur une PIC, savoir qui fait quoi est important — traçabilité et sécurité
* GitLab est l'IAM de fait de la plateforme — tout le monde passe par là
* En croisant les logs GitLab et les access logs des autres outils, on reconstitue le parcours de l'utilisateur
* Usage pratique : détecter un accès anormal, retrouver l'auteur d'une action, auditer les accès

[%notitle]
== Recommandations REX 1

[%step]
* *60 Go/jour* pour un seul XRay en debug — désactiver le debug en prod, systématiquement
* Logs ≠ Métriques — ne pas faire des graphiques avec des logs quand une métrique existe
* Définir la rétention par type de log *avant* de déployer
* Zabbix (métriques) + ELK (logs) + Grafana (centralisation) : des rôles distincts
* Surveiller le gestionnaire de logs — si Logstash tombe en silence, on ne le sait pas
* Timezones, Docker+GELF, auth Kibana — petits détails, grands problèmes

[NOTE.speaker]
====
* Le debug en prod : on a découvert ça en cherchant pourquoi le stockage explosait — 60 Go/jour pour UN seul outil
* Logs ≠ métriques : compter des lignes de log pour "mesurer le trafic" c'est fragile et coûteux — utiliser Prometheus
* La rétention d'abord : sans décision explicite, tout s'accumule indéfiniment
* Timezones : logs avec heures décalées d'une heure pendant des semaines avant de trouver
* Docker + GELF : le container ne démarre pas si le driver GELF ne joint pas Logstash — prévoir un fallback
* Auth : Elasticsearch + Kibana ne peuvent pas être authentifiés en version gratuite → jamais exposer sur internet

Transition : Voilà pour le REX 1. Place au REX 2.
====

[NOTE.speaker]
====
* La rétention d'abord : éviter l'accumulation passive qui coûte cher
