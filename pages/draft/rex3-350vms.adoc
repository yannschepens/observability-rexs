= REX 3 : Monitoring & alerting d'une infra 350 VMs
:imagesdir: assets/default/images

== !

[NOTE.speaker]
====
REX 3 — La mission la plus complexe en termes d'échelle.
350 VMs. Windows, Linux. Cloud, on-prem. Gérées en code et à la main.
Objectif : construire une tour de contrôle — supervision uniforme de l'ensemble du SI.

Transition : Le contexte d'abord.
====

// ─── CONTEXTE & CHALLENGES ───────────────────────────────────────────────────

[%notitle]
== Contexte

Mission d'uniformisation du monitoring chez un client grand compte

[%step]
* ~350 VMs — Debian, CentOS, Windows
* Deux hébergeurs : AWS + on-premise VMWare
* ~200 applications, pilotées par 50 personnes
* AWS géré en code (Terraform + Puppet) — le reste, à la main

[NOTE.speaker]
====
* Le client est un groupe important avec une DSI structurée
* AWS : provisionnement as code, Terraform + Puppet — infra propre
* On-prem : géré manuellement, hétérogène
* Pas de contrainte réseau particulière — pas de réseau isolé, pas de filtrage de flux
* Par contre : pas de lien réseau direct entre AWS et l'on-prem

Transition : Et avant notre arrivée, il y avait déjà du monitoring en place.
====

// ─── L'EXISTANT ──────────────────────────────────────────────────────────────

[%notitle]
== L'existant : deux systèmes qui coexistent

[%step]
* *Prometheus + Grafana* — AWS + on-prem Debian +
  node_exporter auto-déployé, dashboards non unifiés, alertes par équipe sur Slack
* *PRTG* — on-prem uniquement +
  configuré à la main, alertes "au besoin"
* Zéro couverture des services extérieurs (RDS, load balancers, SaaS...)

[NOTE.speaker]
====
* Prometheus est déjà en place et bien installé sur AWS, configuré avec Puppet — c'est une bonne base
* Mais chaque équipe gère ses dashboards et ses alertes de son côté → pas d'uniformité
* PRTG couvre un datacenter on-prem uniquement — géré manuellement, sans vraie stratégie d'alerting
* Résultat : deux systèmes qui ne se parlent pas, une couverture partielle, et des alertes qui partent dans tous les sens

Transition : Avant de proposer quoi que ce soit, on a posé nos convictions.
====

// ─── CONVICTIONS ─────────────────────────────────────────────────────────────

[%notitle]
== Nos convictions

[%step]
* *Monitorer tout le SI avec les mêmes règles : pas réaliste*
* La criticité dépend du rôle, du contexte, de l'impact
* La labélisation est la clé — technique *et* fonctionnelle
* Les outils doivent être simples à installer et à maintenir
* Autodiscovery ou config as code — éviter les opérations manuelles

[NOTE.speaker]
====
* Une alerte CPU à 80% sur un batch de nuit et sur une base de données de production, ce n'est pas la même chose
* Le monitoring ne sert à rien si personne ne le lit — simplifier avant d'enrichir
* L'objectif : éviter les surcharges d'alertes, avoir des dashboards utilisables par tous

Transition : Avec ces convictions, voilà l'approche qu'on a proposée.
====

// ─── APPROCHE ────────────────────────────────────────────────────────────────

[%notitle]
== L'approche : pas de révolution

[%step]
* On garde les technos en place — Prometheus + Grafana
* On garde les métriques existantes — elles suffisent
* On *uniformise la labélisation* — en ajoutant une dimension métier/fonctionnelle
* On définit une *matrice de criticité* — pour l'alerting et les dashboards
* On s'appuie sur l'existant pour déployer les nouveaux labels

[NOTE.speaker]
====
* Pas de révolution technologique, pas de modification de l'existant — c'est un choix délibéré
* Les équipes connaissent Prometheus et Grafana, il ne faut pas les désorienter
* Le vrai travail est dans la labélisation et la stratégie d'alerting — pas dans le remplacement d'outils
* On ajoute : blackbox_exporter pour les sondes HTTP, prtg_exporter pour intégrer PRTG dans Prometheus

Transition : Voyons la stack cible.
====

// ─── ARCHITECTURE & MÉTRIQUES ────────────────────────────────────────────────

[%notitle]
== Stack retenue

[%step]
* *Collecte* : node_exporter (Linux), windows_exporter, blackbox_exporter (HTTP), prtg_exporter
* *Stockage* : Prometheus
* *Alerting* : Alertmanager → Teams + SMS (critiques)
* *Visualisation* : Grafana

//TODO : schéma d'architecture de collecte

[NOTE.speaker]
====
* node_exporter déjà déployé sur Debian par Puppet — et on étend sur CentOS et Windows
* blackbox_exporter : sonde les URLs HTTP/HTTPS des services — code réponse + temps de réponse
* prtg_exporter : fait le pont entre PRTG et Prometheus — centralise tout dans un seul endroit
* Alertmanager : remplace les alertes Slack bricolées par équipe — centralise et orchestre les notifications

Transition : Mais avant l'alerting, il faut parler de labélisation.
====

[%notitle]
== Les métriques à collecter

Le moins possible, mais les bonnes

[%step]
* CPU / RAM / Disque / Réseau — pour les VMs
* Code HTTP + temps de réponse — pour les services
* Métriques spécifiques — pour les services extérieurs (RDS, stockage...)

[NOTE.speaker]
====
* On ne change pas les métriques système — elles sont suffisantes
* On ajoute les sondes HTTP sur tous les services internes et externes
* Pour les services extérieurs : des exporteurs Prometheus existent (RDS, NetApp, etc.)
* Piège : ne pas tout collecter — ça coûte en stockage et noie le signal

Transition : La vraie valeur ajoutée, c'est la labélisation.
====

// ─── LABÉLISATION ────────────────────────────────────────────────────────────

[%notitle]
== La clé : la labélisation

Sans labels métier, pas d'alerting intelligent

[%step]
* `environment` — prod / preprod / dev
* `role` — db, frontend, backend, api, batch, tool, IAM...
* `domain` — équipe/domaine fonctionnel propriétaire
* `project` — application concernée
* `hosting` — aws, on-prem, cloud tiers
* `os` — debian, centos, windows

[NOTE.speaker]
====
* Ces 6 labels permettent de répondre à : qui est impacté ? quelle équipe prévenir ? quelle est la criticité ?
* Le label `role` est le plus structurant — c'est lui qui calibre les seuils d'alerte
* Mise en œuvre : relabelling Prometheus pour les VMs non-AWS, tags AWS directement récupérés via la découverte de services
* Valeur par défaut à `unknown` pour tout label non défini — permet de repérer les trous de couverture

Transition : Avec ces labels, on peut définir une stratégie d'alerting différenciée.
====

// ─── STRATÉGIE D'ALERTING ────────────────────────────────────────────────────

[%notitle]
== Stratégie d'alerting

Le problème n°1 : trop d'alertes = plus d'alertes

[%step]
* Envoyer une notification *uniquement pour une intervention immédiate*
* Les autres alertes : un dashboard de suivi, check tous les matins
* Sévérité : *critical* (action immédiate) / *warning* (traitement dans la journée)
* Zone tampon : 5 minutes avant de déclencher une notification

[NOTE.speaker]
====
* La règle d'or : une alerte qui se déclenche doit appeler une action. Sinon, c'est du bruit.
* Les alertes "warning" n'ont pas besoin d'être notifiées en temps réel — un dashboard suffit
* La zone tampon de 5 minutes élimine les faux positifs liés aux pics courts (batch, démarrage d'appli...)
* Alertmanager gère aussi la déduplication et la répétition des notifications

Transition : Des seuils par rôle, pas des seuils universels.
====

[%notitle]
== Seuils par rôle (production)

//TODO : tableau seuils (extract depuis le pptx)

[%step]
* *Bases de données* — CPU 50/70%, RAM 50/80%, Swap > 0 = alerte immédiate
* *Frontend/API* — CPU 80/90%, temps de réponse HTTP > 500ms/1000ms
* *Batch* — CPU 80/90%, Swap > 50% (tolérance plus haute)
* *IAM/Auth* — CPU 60/80%, temps de réponse > 300ms/800ms

[NOTE.speaker]
====
* Une base de données et un batch n'ont pas les mêmes profils de consommation — inutile d'appliquer les mêmes seuils
* Le swap à 0 sur une DB c'est critique — le même seuil sur un batch c'est normal
* Pour le disque : attention aux volumes alloués — 10% de 2 To = 200 Go, ce n'est pas forcément urgent
* Ces seuils ont été définis avec les équipes — et ils sont faits pour être affinés à l'usage

Transition : Et une fois l'alerte déclenchée, qui reçoit quoi ?
====

[%notitle]
== Canaux d'alerting

La bonne alerte, à la bonne équipe, au bon moment

[%step]
* *Par projet* — canal Teams de l'équipe projet
* *Par domaine* — responsable de domaine (critique uniquement)
* *Par hébergeur* — équipe hébergement concernée
* *Hypercritiques* — SMS en plus (services e-commerce, monitoring lui-même)

[NOTE.speaker]
====
* L'organisation des canaux correspond à l'organisation des équipes — c'est intentionnel
* Un canal Teams par équipe projet, sur abonnement pour les devs, obligatoire pour les leads
* Canal SMS réservé aux services hypercritiques — ne pas banaliser le SMS sinon les gens le coupent
* Power Automate en intermédiaire : c'est lui qui route vers Teams ou SMS selon la criticité

Transition : Et les faux positifs dans tout ça ?
====

// ─── FAUX POSITIFS & DASHBOARDS ──────────────────────────────────────────────

[%notitle]
== Gérer les faux positifs

[%step]
* Labels `unknown` → dashboard dédié pour identifier les VMs mal labelisées
* Maintenance windows — silences Alertmanager pendant les interventions
* Affiner les seuils à l'usage — les premiers réglages ne seront pas parfaits

[NOTE.speaker]
====
* Le dashboard des "unknown" est un outil de qualité de données — il permet de savoir ce qui n'est pas encore couvert
* Les silences Alertmanager permettent de planifier une intervention sans être inondé d'alertes
* Les seuils initiaux sont un point de départ — ils se calibrent sur la durée, pas en une semaine

Transition : Les dashboards pour avoir la vision d'ensemble.
====

[%notitle]
== 3 niveaux de dashboards

[%step]
* *Map* — vue cartographique des services principaux (heartbeat HTTP)
* *Vue globale instances* — état de toutes les VMs, groupées par environnement et projet
* *Détail instance* — drilldown : quelle métrique système est en difficulté ?

[NOTE.speaker]
====
* La map : vue SVG interactive avec des indicateurs de couleur (Grafana Flow Panel) — lisible par les non-techs
* La vue globale : code couleur OK/Warning/Critical — "est-ce que tout va bien ?" en un coup d'œil
* Le drilldown : un clic depuis la vue globale → on arrive sur les métriques CPU/RAM/Disk de la VM concernée
* Même logique pour les services HTTP — vue globale + détail (temps de réponse, code HTTP, certificats)

// ─── RUNBOOK & ESCALADE ──────────────────────────────────────────────────────

[%notitle]
== Runbook & escalade

Une alerte sans runbook, c'est du bruit habillé

[%step]
* Chaque alerte P1/P2 → un runbook (même minimal)
* Escalade explicite : qui appelle qui, dans quel ordre, avec quelles infos
* Silences Alertmanager pour les maintenances planifiées
* Challenger les alertes régulièrement — si aucune action en 3 mois, la supprimer

[NOTE.speaker]
====
* Le runbook n'a pas besoin d'être exhaustif — "regarder ce dashboard en premier, vérifier ce process" suffit
* L'escalade : sans règle écrite, chacun improvise. On a défini 3 niveaux :
  ** L1 : équipe projet — alerte warning → canal Teams
  ** L2 : responsable de domaine — alerte critique → Teams + autre canal
  ** L3 : hébergement + DSI — service hypercritique hors ligne → SMS
* Les maintenances planifiées : le silence Alertmanager doit être posé AVANT l'intervention, pas après
* Challenger les alertes : si une alerte ne déclenche jamais d'action, c'est soit un seuil trop bas, soit une alerte inutile

Transition : Pour finir, quelques conseils.

[%notitle]
== Conseils pratiques REX 3

[%step]
* Inventaire d'abord — on ne monitore pas ce qu'on ne connaît pas
* Labélisation = fondation — sans elle, pas d'alerting différencié
* Garder les technos en place — l'adoption compte plus que l'outil parfait
* Moins d'alertes, mieux ciblées — la qualité prime sur la quantité
* Monitorer le système de monitoring — qui surveille le surveillant ?

[NOTE.speaker]
====
* L'inventaire : on a passé du temps à cartographier les VMs et les services avant de toucher à quoi que ce soit
* La labélisation : c'est le travail qui paie sur la durée — chaque nouvelle VM doit être labélisée dès le déploiement
* Les outils en place : les équipes connaissent Prometheus et Grafana, inutile de les changer pour changer
* Le monitoring du monitoring : si Prometheus tombe, plus personne ne voit rien — c'est la première alerte à mettre en place
