// ─── ÉTAPE 1 : ANALYSE ───────────────────────────────────────────────────────

[%notitle]
== Étape 1 — Le grand tri

Explorer, lister, mesurer

[%step]
* Inventaire complet dans un fichier Excel
* Volumétrie réelle par index
* Pertinence, catégorie, durée de conservation
* Suppression des logs inutiles

[NOTE.speaker]
====
* Step : On liste tout — l'équivalent de sortir tous les vêtements de l'armoire sur le lit
* Step : Pour chaque index : taille, nombre de docs
* Step : challenge de la pertinence, qui l'utilise
* Step : On ne supprime que ce qui est clairement inutile — pas de prise de risque à ce stade

Transition : Déjà sur cette étape le résultat parle de lui-même.
====

[%notitle]
== Résultat du grand tri

[.big-number]
782 Go → 638 Go

* Firewall : 80 Go → 60 Go (−25%)
* VMWare : 140 Go → 80 Go (−43%)
* *Rien qu'en supprimant ce qui ne devrait pas être là*

[NOTE.speaker]
====
Lire les valeurs

* Ces gains sont obtenus avant de toucher à la configuration
* Juste en ouvrant l'armoire et en jetant ce qui est clairement inutile
* Ça illustre bien l'état des lieux initial — des logs de DEBUG vieux de plusieurs mois, personne ne les lisait

Transition : Étape 2, catégoriser ce qu'on garde.

// ─── ÉTAPE 2 : CATÉGORISATION ────────────────────────────────────────────────

[%notitle]
== Étape 2 — Catégorisation

Chaque log à sa place, chaque place à son log

[%step]
* Découper chaque index selon :
** *Type* — system, appli, réseau, métier...
** *Niveau* — error, warn, info, debug
** *Conservation* — légale, opérationnelle, aucune
** *Débit* — volumétrie par unité de temps

[NOTE.speaker]
====
Le but de cette étape est d'y voir plus clair et surtout de pouvoir, par la suite, définir des règles par "catégorie"

* On trie par catégories — comme les vêtements par couleur et par matière
* Faire passer les steps en lisant

* Résultat concret : l'index des logs système éclate en 5 datastreams

Transition : La catégorisation permet de définir une vraie stratégie de stockage.
====

// ─── ÉTAPE 3 : STOCKAGE ──────────────────────────────────────────────────────

[%notitle]
== Étape 3 — Ranger intelligemment

Les logs froids à la cave

[%step]
* Ajout de *métadonnées* sur les logs (ECS Elasticsearch)
* *Datastreams* par catégorie
* *ILM* (Index Lifecycle Management) par datastream
* hot → warm → cold → delete

[NOTE.speaker]
====
Ici, on va réflechir à ranger les logs selon deux dimensions : où et quand

* Step : On va déjà ajouté les méta données basée sur nos catégories en suivant le standard ECS
** Dans logstash
* Step : Ensuite on utiliser la technologie des datastreams : index virtuelle au dessus de plusieurs index physique. Ce qui débloque les
* Step : ILM = Index Lifecycle Management. On peut définir des règles de gestions des indexs dans Kibana, par exempke
** Step : Les logs chauds restent accessibles rapidement (hot), les logs froids sont compressés sur des nœuds moins chers (warm/cold), et expirés automatiquement (delete)
* Et donc, chaque datastream a sa politique de cycle de vie — les logs d'accès restent 1 an, les logs d'erreur 1 mois
* Résultat inattendu : meilleure compression grâce au mapping structuré
  * Logs proxy : 30M entrées pour 60 Go → 45M entrées pour moins d'espace
  * Hypothèse : meilleure compression liée à la cohérence des champs

Transition : Une fois les datastreams et les règles en place, on améliore la config Logstash.
====

// ─── ÉTAPE 4 : LOGSTASH ──────────────────────────────────────────────────────

[%notitle]
== Étape 4 — Logstash : la config statique, c'est une dette

[%step]
* Analyse de la configuration existante
* Ajout des métadonnées dans les filtres
* *Variabilisation* des outputs basée sur les métas
* Sécurisation des mots de passe

[NOTE.speaker]
====
Ici, on est surtout sur une histoire de flexibilité et de faciliter de maintenance de code.

* On passe les steppers
* La configuration Logstash était statique : un output codé en dur par destination
* En variabilisant les outputs sur les métadonnées, on dirige les logs vers le bon datastream automatiquement
* C'est comme étiqueter les rayons de l'armoire — plus besoin de chercher

Transition : Et pour finir, deux autres sujets étudiés mais pas encore déployés.
====

// ─── ÉTAPES 5 & 6 ────────────────────────────────────────────────────────────

[%notitle]
== Étapes 5 & 6 — Agrégation & archivage

Étudiés, pas encore déployés

[%step]
* *Agrégation* : downsampling, rollup ou transformation +
→ non mis en place, contrainte de ne pas perdre de données
* *Archivage* : snapshot Elasticsearch ou export manuel +
→ présenté, budgété pour une prochaine phase

[NOTE.speaker]
====
Deux sujets qu'on a vu, mais que je n'ai pas déployé à l'époque, je pense que c'est fait aujourd'hui.

* Step : Agrégation : l'étape suivante — stocker des résumés au lieu du détail pour les logs anciens
* Step : Archivae : les logs à conservation légale doivent pouvoir être retrouvés facilement — ils vont au fond du placard mais pas à la poubelle

[%notitle]
== 🧺 Take-away REX 2

"La meilleure donnée, c'est celle qu'on n'a pas besoin de stocker."

[%step]
* Faire l'inventaire *avant* de toucher à quoi que ce soit
* Se poser la question Marie Kondo pour chaque flux : décision opérationnelle ou non ?
* ILM dès le départ — les logs froids à la cave, pas dans le même tiroir que les chauds
* Variabiliser Logstash — finir les outputs codés en dur
* La sobriété, c'est la survie de votre stack ELK

[NOTE.speaker]
====
Donc, la conclusion à emporter

* On passe les steps

* L'inventaire Excel est peu glamour — mais c'est la base de tout le reste
* ILM : sur ce projet, on avait des index sans aucune politique de suppression depuis des années
* La sobriété des logs n'est pas une contrainte technique — c'est une discipline d'équipe
