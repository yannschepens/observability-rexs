== Pourquoi monitorer ?

[%step]
* Traçabilité
* Disponibilité
* Sécurité / Légalité

[NOTE.speaker]
====
* Traçabilité : suivre ce qui se passe sur la plateforme — Qui ? Quand ? Où ?
* Disponibilité : s'assurer que les plateformes et les outils fonctionnent correctement
* Sécurité : le monitoring protège l'infra. Et il y a un aspect légal — certains logs doivent être conservés.

[%notitle]
== Le tas de linge

* 120 types de log
* Quelques gigas par jour
* JSON, Combined Apache, Pipe separator...
* Fichiers et sortie standard Docker
* RGPD, légalité, rétention variable

[NOTE.speaker]
====
120 types de log, des gigas par jour, des formats différents selon chaque outil.
C'est le panier de linge en vrac.

* Certains logs doivent être conservés — les logs d'accès par exemple
* D'autres sont à anonymiser — RGPD
* Certains sont inutiles à garder au-delà d'un mois

Bref, la première question n'est pas "comment les stocker" — c'est "lesquels méritent d'exister".

== Nettoyage, parse & enrichment

"Séparer le coton du synthétique"

[NOTE.speaker]
====
C'est l'étape la plus importante — et la plus négligée.
Si tu envoies tout brut dans Elasticsearch, tu as stocké du bruit, pas de la donnée.
====

[%notitle]
== Agrégateur : Logstash

image::rex1-logstash.png[1, 50%]

[NOTE.speaker]
====
Problématique : pour normaliser les logs, il faut d'abord les centraliser.

Solution : Logstash — input / filter / output.

Laisser les logs tels quels, en mode "log sink", c'est comme mettre tout le linge dans un seul tiroir : introuvable.

Transition : Donc on normalise.
====

[%notitle]
== Logstash : normalisation

[%step]
* Découpe des logs textes en JSON
* Taggage des types de logs
* Taggage de l'origine des logs
* Ajout d'informations (localisation via GeoIP)
* Normalisation des logs d'accès HTTP
* Organisation par mois et année
* Anonymisation RGPD

[NOTE.speaker]
====
C'est ici qu'on sépare le coton du synthétique.

* On étiquette chaque log : type, origine, environnement
* Un log sans étiquette, c'est une chaussette orpheline — inutilisable
* Le GeoIP enrichit chaque accès avec la localisation → utile pour détecter des accès anormaux
* L'anonymisation RGPD est appliquée à ce niveau — une fois en entrée, pas à chaque requête

[%notitle]
== 👕 Take-away REX 1

"Un dashboard sans parsing propre, c'est juste du bruit en couleur."

[%step]
* *60 Go/jour* pour un XRay en debug — désactiver le debug en prod
* Logs ≠ Métriques — ne pas faire des graphiques avec des logs quand une métrique existe
* Étiqueter chaque log dès la collecte — type, origine, env
* Rétention par type dès J1 — sinon tout s'accumule
* Surveiller le gestionnaire de logs — si Logstash tombe en silence, on ne le sait pas

[NOTE.speaker]
====
* L'image à retenir : séparer le coton du synthétique — sans tri, la machine à laver ne fait qu'homogénéiser le chaos
* Le debug en prod : 60 Go/jour pour UN seul outil. Ça semble anodin, ça ruine le stockage.
* Logs ≠ métriques : compter des lignes de log pour mesurer le trafic c'est fragile — Prometheus le fait mieux
* Timezones, Docker+GELF, auth Kibana : petits détails, grands problèmes

Transition : Place au REX 2 — l'armoire qui déborde.
