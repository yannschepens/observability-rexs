* Quelques gigas par jour

** Certains logs doivent être conservés — les logs d'accès par exemple

** Certains sont inutiles à garder au-delà d'un mois

* Bref, la première question n'est pas "comment les stocker" — c'est "lesquels méritent d'exister".

// ─── NETTOYAGE / PARSE & ENRICHMENT ─────────────────────────────────────────

== Nettoyage, parse & enrichment

"Séparer le coton du synthétique"

[NOTE.speaker]
====
* C'est l'étape la plus importante — et souvent la plus négligée.
* Si tu envoies tout brut dans Elasticsearch, tu as stocké du bruit, pas de la donnée.

Transition : et pour ça, on va utilisé un agrégateur
====

[%notitle]
== Agrégateur : Logstash

[cols="1,2",frame=none,grid=none]
|===
a|
image::rex1/rex1-logstash.svg[1, 200%]
a|
[%step]
* Découpe des logs textes en JSON
* Taggage des types et origine de logs
* Ajout d'informations (localisation via GeoIP)
* Normalisation des logs d'accès HTTP
* Organisation par mois et année
|===

[NOTE.speaker]
====
* Problématique : pour normaliser les logs, il faut d'abord les centraliser.
* Solution : Logstash — input / filter / output.
* Laisser les logs tels quels, en mode "log sink", c'est comme mettre tout le linge dans un seul tiroir : introuvable.
* Donc, on normalise : C'est ici qu'on sépare le coton du synthétique.

Lis les steppers, puis

* On étiquette chaque log : type, origine, environnement
* Un log sans étiquette, c'est une chaussette orpheline — inutilisable
* Le GeoIP enrichit chaque accès avec la localisation → utile pour détecter des accès anormaux
* L'idéal qu'on a pas pu faire : l'anonymisation RGPD est appliquée à ce niveau — une fois en entrée, pas à chaque requête

Transition :  C'est bien on a nos logs, mais on les stocke où ?
====

== Stockage

[.badge]
image::rex1/rex1-es.svg[1, 200%]

[.main-image]
image::rex1/rex1-indexes.png[1]


[NOTE.speaker]
====
* On a choisi Elasticsearch — reconnu dans le domaine, gère très bien le JSON.

* Mais ça ne se fait pas tout seul non plus, il faut réfléchir à la stratégie de stockage

Format des noms d'index : `<service>_<env>-<log_file>-YYYY-MM`

* Une centaine d'index par mois
* Rétention via script selon le type :
** 1 an pour les logs d'accès
** 1 mois pour les logs d'erreurs
** 1 mois pour les logs applicatifs

Note : aujourd'hui il y a les ILM et les datastreams ELK — ce n'était pas le cas à l'époque.
